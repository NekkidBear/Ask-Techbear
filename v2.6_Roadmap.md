# Ask TechBear v2.6 Roadmap — Evaluation, Reporting, and Memory Controls

_Gymnarctos Studios LLC_
_Branch: `feature/v2.6-evaluation-reporting`_
_Base tag: `v2.5-async-baseline`_

---

## Purpose

v2.5 established the async pipeline baseline: moderation, factual generation, critique layers, voice rewrite, semantic checks, character checks, editorial review, handoff artifacts, and benchmark execution.

v2.6 turns those pipeline outputs into a persistent evaluation system.

The goal is not to add more model complexity yet. The goal is to capture, review, score, compare, and analyze pipeline behavior over time.

---

## v2.6 Goals

### Primary Goals

- Persist pipeline runs in PostgreSQL
- Store LLM-generated scores and critique outputs
- Store human review scores and notes
- Track human-vs-LLM score deltas
- Generate human review workbooks
- Generate human-readable benchmark reports
- Support calibration of judges, prompts, moderation, and retrieval

### Secondary Goals

- Add lore-recall testing support
- Improve session context tracking for repeated jokes/metaphors
- Preserve anti-regurgitation evidence
- Prepare for future corpus re-ingestion workflows

---

## Non-Goals

v2.6 should not introduce major new model behavior unless required for evaluation storage.

Deferred:

- New model selection
- LoRA/fine-tuning
- Live UI overhaul
- WebSocket/SSE response delivery
- STT post-performance transcription
- Full human review dashboard
- Production publishing workflow
- Automated corpus re-ingestion

## Core Schema Additions

### `pipeline_runs`

Stores one execution of the pipeline against one question.

Recommended fields:

```sql
id UUID PRIMARY KEY
question_id INTEGER NOT NULL REFERENCES questions(id)
pipeline_version VARCHAR NOT NULL
run_label VARCHAR
status VARCHAR NOT NULL
started_at TIMESTAMPTZ
completed_at TIMESTAMPTZ
```

Purpose:

```text
One question can have many pipeline runs.
Example:
corpus_001 → v2.5 result
corpus_001 → v2.6 result
corpus_001 → v2.7 result
```

## Context and Memory Work

### Session Context

Track recent output patterns to reduce repetition across a live or async batch.

Candidate fields:

```text
recent_openers
recent_metaphors
recent_jokes
recent_source_posts
recent_voice_chunks
recent_lore_references
```

Purpose:

```text
Avoid repeating:
- "I'm not sure whether to laugh or cry"
- "paper umbrella in a hurricane"
- "I nearly blue-screened"
- "sequins in a magnet factory"
```

---

### Conversation Context

Track attendee-specific follow-up context.

Future fields:

```text
conversation_id
conversation_depth
parent_question_id
previous_questions
previous_answers
```

Purpose:

```text
Q1: Have you ever met Captain Janeway?
Q2: Did she get her coffee fixed?
```

The second question requires per-attendee context.

---

### Lore Retrieval Context

v2.6 should document and begin preparing for lore-aware retrieval.

Lore questions should query fiction/canon chunks rather than treating fiction as factual support.

Examples:

```text
Have you ever met Captain Janeway?
What happened in the Delta Quadrant?
Did TechBear ever visit Discworld?
What was the Jurassic Park incident?
```

Expected retrieval mode:

```text
retrieval_mode = lore
is_fiction = true
```

## Known Risks

### Schema Drift

The project currently relies on SQLAlchemy `create_all()`. v2.6 introduces more complex data persistence. Alembic migrations may become necessary soon.

### Overfitting to Small Test Sets

The first 11-question benchmark is useful but too small for broad conclusions. Target 30–50 reviewed examples before major judge recalibration.

### Corpus Contamination

Do not immediately ingest every corrected answer into the corpus. Store feedback separately until reviewed and approved as canonical.

### Lore vs Fact Confusion

Fictional TechBear stories are valid for voice and lore, but not factual technical support. Retrieval mode must respect that distinction.

### Judge Overconfidence

Some judges may score confidently while missing key editorial or lore failures. Human deltas are the calibration signal.

## v2.6.1 Roadmap Addendum — Lore Retrieval Architecture

## New Objective

Introduce a dedicated lore retrieval subsystem to preserve TechBear canon while preventing fictional stories from contaminating factual technical guidance.

This work is driven by benchmark observations:

- Factual retrieval is functioning.
- Voice retrieval is functioning.
- Lore recall is inconsistent.
- Canonical TechBear stories are not always retrieved when referenced directly.

Example:

Question:

"Have you ever met Captain Janeway?"

Expected behavior:

Retrieve and synthesize content from:

- Coffee Crisis in the Delta Quadrant
- Captain Janeway
- Tom Paris
- Talaxian tomato replicator incident

Observed behavior:

Model recognized Janeway as a Star Trek character but failed to retrieve TechBear-specific canon.

This indicates a lore retrieval failure rather than a generation failure.

---

## Retrieval Architecture v2.6

### Collection: techbear_facts

Purpose:

Real-world technical information.

Sources:

- Ask TechBear technical articles
- Knowledge corpus
- Technical reference material

Rules:

- is_fiction = false
- Used only for factual support

---

### Collection: techbear_voice

Purpose:

Character voice, style, humor, pacing, and personality.

Sources:

- Published Ask TechBear responses
- Character files
- Fiction and nonfiction articles

Rules:

- May include fiction
- Not used as factual support

---

### Collection: techbear_lore

Purpose:

Persistent character canon.

Sources:

- Multiverse stories
- Voyager stories
- Discworld stories
- Jurassic Park stories
- Rocky Horror / Transylvania stories
- Other recurring narrative continuity

Rules:

- is_fiction = true
- Never used for factual validation
- Used only for lore and character-memory questions
